# Design — 测速 & DNS 移动端 iOS-native v5 重构

## Architecture overview

All work lands inside `worker.js`:

- **CSS** added to the `CSS_COMMON` template literal at the very end of the
  existing `@media (max-width: 768px)` v5 block (worker.js:1453+). One new
  appended block `=== Mobile v5 — 测速 & DNS specialist ===`.
- **HTML** edits inside the `#sec-speed` section (worker.js:2258–2364):
  inject the segmented control, latency-bar markup, overflow trigger, and
  selection-count action bar. Existing IDs preserved.
- **JS** appended to the existing inline `<script>` block(s) in `#sec-speed`:
  three small drivers — segmented control sync, overflow sheet open/close,
  selection-count badge update. No new external dependencies.

No new files, no build step changes, no API contract changes.

## Component contracts

### 1. `.sd-dns-card` — Current effective DNS card (R1)

```
<div class="sd-dns-card" id="dnsStatusCard">
  <div class="sd-dns-head">
    <span class="sd-eyebrow">DNS · LIVE</span>
    <span class="sd-domain" id="sd-domain-label">—</span>
  </div>
  <ul class="sd-dns-list" id="dnsStatus">
    <li class="sd-dns-row">
      <span class="sd-rec-pill is-A">A</span>
      <code class="sd-ip">104.16.124.x</code>
      <span class="sd-geo">CN · 电信</span>
    </li>
    ...
  </ul>
</div>
```

- Renders by JS (`renderDnsStatus`) on mobile **and** desktop — the existing
  `#dnsStatus` element is reused; new inner markup is generated by JS. On
  desktop, the surrounding `.sd-dns-card` chrome is suppressed (`display:
  contents`) so the desktop visual stays unchanged.
- Record-type pill classes: `.is-A` (primary-soft), `.is-AAAA` (cyan accent),
  `.is-CNAME` (warn-soft) — reuses existing soft-color tokens.

### 2. `.sd-isp-seg` — ISP segmented control (R2)

```
<nav class="sd-isp-seg" role="tablist" aria-label="ISP 筛选">
  <button role="tab" data-value="all" aria-selected="true">综合</button>
  <button role="tab" data-value="电信">电信</button>
  ...
</nav>
```

- Mobile-only. Native `<select id="ipType">` is `display:none` on mobile,
  retained for desktop and as the source-of-truth for JS handlers
  (`fetchRemoteAndTest` reads `document.getElementById('ipType').value`).
- Click handler: `ev.currentTarget.dataset.value → ipType.value = value;
  ipType.dispatchEvent(new Event('change'))`. Updates `aria-selected`.
- Horizontally scrollable via `overflow-x: auto; scroll-snap-type: x mandatory`.
  Scroll into view the selected segment on init.

### 3. `.sd-action-stack` — Primary CTA + overflow (R3)

<div class="sd-action-stack">

- Existing toolbar buttons remain in the DOM but `display:none` on mobile.
- A new `#sdMoreSheet` is added at the bottom of `#sec-speed`, mirroring the

- Selection-count action-bar `#sdSelectionBar`:
  ```

  ```
  Driven by a single `updateSelectionBar()` called on every row checkbox
  change + selectAll. Slides up from below the card (CSS `transform`).

### 4. `.sd-custom-fold` — Collapsible custom source (R4)

- Uses native `<details>` for keyboard/AT support. CSS hides the default

### 5. `.sd-node-card` — Node row (R5)

The current `<tr>` markup is preserved — CSS on mobile re-styles each `<tr>`
as a single card, and each `<td>` as a block within it, **without** the
existing `td::before` data-label labels (those are suppressed via a more
specific selector inside the new block).

Layout uses CSS Grid:

```
.app-section[data-section="speed"] tbody tr {
  display: grid;
  grid-template-areas:
    "check ip   chip"
    "bar   bar  bar"
    "stat  acts acts";
  grid-template-columns: auto 1fr auto;
  gap: var(--space-2);
  padding: var(--space-3) var(--space-3-5);
}
.app-section[data-section="speed"] tbody td { display: contents; }
```

Each `<td>` is `display: contents` so its children become direct grid items,
slotted via `grid-area` set per `data-label`:

```
td[data-label="勾选节点"] > * { grid-area: check; }
td[data-label="专属节点"] > * { grid-area: ip; }
td[data-label="记录/归属地"] { grid-area: chip; } /* this one keeps the td as flex */
td[data-label="预估延迟"] { grid-area: bar; }
td[data-label="连通状态"] { grid-area: stat; }
td[data-label="快捷操作"] { grid-area: acts; }
```

**Latency bar** is rendered by augmenting the existing `.latency` cell. JS that
sets `.latency` text gets a small wrapper: on mobile, the text is moved into
a `<span class="sd-lat-val">` and a sibling `<span class="sd-lat-bar">` with
10 child cells is generated by a tiny `applyLatencyBar(td)` helper, called
from the existing post-test code paths after `data-ms` is set.

To avoid touching every speedtest call site, we use a `MutationObserver` on
`#testTableBody` watching `data-ms` attribute changes; on any change, the
observer recomputes the bar/color. Cheap (~10 nodes) and additive.

Color thresholds:
- `--sd-lat-ok` = `<150` → `var(--ok)`
- `--sd-lat-warn` = `<400` → `var(--warn)`
- `--sd-lat-err` = `≥400` → `var(--err)`
- `9999` (loading) → uses `.skeleton` shimmer on bar cells.

### 6. 优选 CDN 域名 card (R6)

`renderOptimizedDomains` is **modified** to emit `data-label` attrs on each
`<td>` (it already does for most, verified at worker.js:2425–2436 — currently
NO `data-label` set; we add them). Then CSS rewrites the `<tr>` on mobile as
a card with grid layout similar to node-card:

```
"domain   toggle"
"note     note"
"ms-chip  actions"
```

Actions row: 替换DNS as the primary action (full pill width minus delete icon
on the right when not builtin). The `dnsReadyHint` text duplicates the
disabled-button tooltip in a muted line under the action toolbar.

### 7. Section header (R7)

Adds a `.ios-page-header` block at the top of `#sec-speed`, mobile-visible only:

```
<header class="ios-page-header sd-page-header">
  <h1 class="ios-large-title">测速 &amp; DNS</h1>
  <p class="ios-large-sub">节点延迟与解析探测</p>
</header>
```

Existing in-card `.section-title` `<h2>`s are `display:none` on mobile inside
`#sec-speed`, since the page header now carries the title context.

## Data flow

```
existing JS                     new mobile shim                  DOM
-----------                     ---------------                  ---
fetchRemoteAndTest()  ─────►   (unchanged)                ──►  testTableBody
   row HTML strings              applyLatencyBar()        ──►  .sd-lat-bar cells
   set td.data-ms                 ▲ via MutationObserver
                                  │
selectAll / row checkbox ─────►  updateSelectionBar()    ──►  #sdSelectionBar
                                  ▲ via change listener delegated on testTableBody

segmented click       ─────►   ipType.value = …               (existing handlers
                                  ipType.dispatchEvent          read ipType.value)

更多 trigger          ─────►   openSdMoreSheet()         ──►  #sdMoreSheet visible
                                  routes to existing fns       (sheet buttons reuse
                                                                onclick handlers from
                                                                the original menu)
```

No state is duplicated. The native `<select>` and original buttons remain the
source of truth; the mobile shims are pure view-layer.

## Boundaries / tradeoffs

- **Why grid + `display: contents` over rebuilding HTML?**
  Touching the row-rendering JS would mean editing every speedtest fan-out
  (`fetchRemoteAndTest`, `testCustomIPs`, `fetchCustomApiAndTest` at
  3846/3881/3919). High risk of regressions in untested paths. Keeping the
  `<tr><td>` markup and reshaping it with CSS Grid + `display: contents` is a
  pure CSS retrofit; only the latency-bar augmentation needs a tiny JS shim
  (MutationObserver), which is contained to one place.

- **Why MutationObserver instead of patching JS?**
  Same reason — minimizes code touch. The observer is local to `#testTableBody`
  and only runs on `data-ms` attribute changes (filtered by `attributeFilter`).
  Cost ≈ one callback per tested IP, ≤30 IPs per run.

- **Why `<details>` for custom source?**
  Native, keyboard-accessible, AT-friendly, no JS, animates via CSS via
  height transitions on the inner body (not the `<details>` itself).

- **Compatibility**: All targeted properties (`display: contents`,
  `grid-template-areas`, CSS variables, `backdrop-filter` already in use,
  `<details>`, scroll-snap) are supported in iOS Safari ≥ 14 and Android
  Chrome ≥ 90. The app already targets the same baseline (existing v5 CSS
  uses `backdrop-filter`, CSS Grid, `:has()` is **not** required here).

- **Rollback shape**: The new CSS block is a clearly fenced
  `/* === Mobile v5 — 测速 & DNS specialist === */` … `/* === /Mobile v5 ===
  */` region; the new HTML additions are clearly fenced; the new JS shim is
  one fenced block. Reverting = deleting those three fenced regions and the
  added `data-label` attributes in `renderOptimizedDomains`. No data
  migrations.

## Risks

| Risk | Mitigation |
|---|---|
| Desktop visual regression from CSS leaking out of MQ | All new rules wrapped in `@media (max-width: 768px)` from the start; verified by re-loading the desktop page side-by-side after each change. |
| `display: contents` accessibility bug in older Safari | The actual interactive children (`input`, `button`, `code`) retain their semantics; only the `<td>` is `display: contents`. Tested pattern, widely used. |
| MutationObserver firing on unrelated row updates | `attributeFilter: ['data-ms']` + subtree: true, callback short-circuits if `data-ms` unchanged or row has no `.latency` cell. |
| 选中计数 bar overlapping bottom Tab Bar | `bottom: calc(72px + env(safe-area-inset-bottom) + var(--space-3))` so it sits above the tab bar. |
| iOS scroll-anchoring on segmented scroll | `scroll-snap-type: x mandatory` + `overscroll-behavior-x: contain`. |
| Dark-mode contrast of latency-bar empty cells | Use `var(--ios-fill-quat)` (already dark-aware) for unfilled cells. |
| 优选CDN render path overwrites our `data-label` attrs | We're modifying `renderOptimizedDomains` itself to emit them — single source. |

## Out of scope

- 3xx 重定向白名单 card (PRD non-goal).
- Section navigation chrome (`#mobileTopbarCompact`, bottom tab bar) — already shipped in v2.4.0.
- New speedtest features, new APIs, new endpoints.
